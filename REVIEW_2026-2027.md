### Co-requisite / prerequisite mis-parse — bulk fix (April 2026)

Chris's stress-test surfaced a systematic data bug: during the bake
from the catalogue PDF, several "Recommended" entries were transcribed
as **Co-requisites** or **Pre-requisites** in the CSV. Examples he
flagged: PHY3001 (Quantum Mechanics) listed MAT2006 as a co-req when
the catalogue has MAT2006 under *Recommended*; PRA2019 (Natural
Science Illustration) listed PRA2015 as a co-req when it's actually
just a *Recommendation*.

Audited all four period CSVs against the catalogue markdown and
flipped **18 rows** where the CSV asserted a requirement but the
catalogue's first bullet in that section says "None":

**Co-requisites flipped to FALSE (12):** PHY3001, PHY3009, PHY2012,
PRA2019, PRA3503 (P1) · PHY3004 (P2) · BIO2005, CHE3011 (P4) ·
CHE3012, PHY2005 (P5) · PHY2006, PRA2006 (P1, see above).

**Prerequisites flipped to FALSE (6):** INT3002 (P1) · BIO3010 (P4) ·
PHY2007, INT2009, PRA2009, PRA2023 (P5).

The *Recommended* info is still visible in the full catalogue view —
it just no longer surfaces as a fake warning in the planner summary.

**Remaining ambiguous cases worth a human eyeball:**

- CHE2010 ↔ PRA2034 (P5): cross-referenced as co-reqs in the CSV.
  Catalogue uses `•` bullets (no ✓ prefix) so my auto-audit flagged
  them; confirmed via the P5 Frame Schedule PDF that the ◊ markers
  line up — the pair is legitimate. Left untouched.

**Multi-code co-requisites (now supported):**

PRA1101 and PRA1102 have catalogue-declared co-reqs with two codes each:

- PRA1101 (P1) → `CHE1101 + PHY1101`
- PRA1102 (P2) → `BIO1101 + INT1101`

The `CorequisiteOf` field previously only held a single code, so these
co-reqs were under-specified (a student picking PRA1102 + BIO1101 +
something-else would pass the check even though INT1101 is missing).

The schema now accepts the same `+` (AND) / `/` (OR) grammar as
`PrerequisiteCodes`. The CSV values above have been updated, and a
new `checkCoreqMet(expr, presentCodes)` helper in the planner
evaluates them against the set of currently-selected codes +
transcript-passed codes (auto-waive). All existing single-code co-reqs
continue to work unchanged (a single code is just a degenerate case
of the expression grammar).

A one-off audit script at bake time that cross-checks every
`Corequisite=TRUE` / `Prerequisite=TRUE` against the catalogue would
prevent this class of error from re-entering.

For practicals **already in the 2025-2026 P5 CSV**, I reused their known-good
`Days` values. For practicals **only in P1 / P2 / P4** (or new this year),
the frame schedule gives only the day of the week; I used the conservative
default of both AM and PM per listed day. This may cause over-reporting of
conflicts in rare cases. Give me AM/PM specifics when you have them and
